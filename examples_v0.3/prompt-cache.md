## 第 1 帧：先别管术语

假设你已经和 AI 聊了好几轮。

```
你：我是一名程序员
AI：你好！作为程序员，有什么我可以帮你的吗？
你：帮我写个 Python 函数，计算数组的平均值
AI：def calculate_average(arr):
        return sum(arr)/len(arr)
你：如果数组是空的怎么办？
AI：好问题！我们需要加一个检查……
```

现在你要问下一个问题：**"那如果数组里有非数字呢？"**

先别管"prompt cache""KV cache""前缀匹配"这些词。

我们要搞清楚的，其实就是一件事——**AI 要回答你最后这个问题，需不需要重新读你们之前所有的对话？**

> 用户回复：应该不需要吧？

对。直觉是对的。但"应该不需要"和"系统真的不会重复算"之间，还有一段距离。

## 第 2 帧：朴素做法怎么坏

很多系统如果没做特殊设计，确实会傻乎乎地每次都把全部历史从头处理一遍。

```
你的历史对话（从第 1 条到第 3 条）
┌─────────────────────────────────────┐
│ 1. 我是一名程序员                     │
│ 2. 帮我写个 Python 函数，计算数组平均值 │
│ 3. 如果数组是空的怎么办？              │
└─────────────────────────────────────┘
                    ↓
              你要发的第 4 条
      "那如果数组里有非数字呢？"
                    ↓
        ┌───────────────────┐
        │   AI 的处理流程     │
        │ 1. 读第 1 条 → 处理 │
        │ 2. 读第 2 条 → 处理 │
        │ 3. 读第 3 条 → 处理 │
        │ 4. 读第 4 条 → 处理 │
        │  （全部重来一遍）  │
        └───────────────────┘
```

问题在哪？

第 1、2、3 条你已经聊过，AI 当时已经"理解"过一遍了。但下次你再发新消息，它又从第 1 条开始重新读、重新算。

就像你每节课都问老师一个旧问题，老师都要从"1 + 1"重新教一遍。

这会带来两个麻烦：

- **慢**：每次都要处理全部历史，越聊越卡
- **贵**：按 Token 收费的 API，每次都在为同样的内容重复付钱

## 第 3 帧：聪明办法——把"已经处理过的部分"存起来

那 AI 能不能像人一样，**把之前处理过的对话"记住"，下次直接用**？

```
第 1 轮对话
┌─────────────────┐
│ 1. 我是一名程序员 │ → AI 处理完，记下来
└─────────────────┘

第 2 轮对话
┌─────────────────────────────────┐
│ 1. 我是一名程序员                 │ ← 已处理，不用再算
│ 2. 帮我写个 Python 函数……        │ → AI 只处理新内容
└─────────────────────────────────┘

第 3 轮对话
┌─────────────────────────────────────────┐
│ 1. 我是一名程序员                         │ ← 已处理
│ 2. 帮我写个 Python 函数……                │ ← 已处理
│ 3. 如果数组是空的怎么办？                  │ → AI 只处理新内容
└─────────────────────────────────────────┘

第 4 轮对话（你问新问题时）
┌─────────────────────────────────────────────────┐
│ 1-3 条：已经处理过，存在"记忆"里                   │
│ 4. 那如果数组里有非数字呢？                        │ → AI 只处理这条新消息
└─────────────────────────────────────────────────┘
```

这种**把已经处理过的历史对话缓存起来，下次直接复用**的机制，就是 **Prompt Cache（提示缓存）**。

---

## 第 4 帧：它到底缓存了什么？

在大语言模型里，处理一段文本时，模型内部会产生很多中间计算结果（比如每一层注意力机制算出来的数值）。这些中间结果才是真正耗时、耗资源的部分。

Prompt Cache 就是把这些**中间计算结果**存起来。下次遇到相同的前缀（比如同样的历史对话），就不用重新算了，直接把缓存的中间结果拿出来用。

```
没有缓存：
输入文本 → [重新计算每一层] → 输出

有缓存：
输入文本 → [发现前缀已缓存] → [只算新部分] → 输出
                 ↓
          直接复用缓存的中间结果
```

## 第 5 帧：代码补全场景——为什么这事特别重要

你在 IDE 里写代码，每敲一个字符，Copilot 都在背后"思考"该给你什么建议。这个过程必须极快（通常 < 200 毫秒），否则体验就卡了。

但你的代码文件可能很长，而且你每次只改了一小部分。

```
你敲了一个字符
        ↓
Copilot 需要：读取整个文件 + 历史上下文 + 你刚敲的这个字符
        ↓
    重新计算全部 → 太慢了！
```

有 Prompt Cache 之后：

```
你敲了一个字符
        ↓
Copilot 发现：文件前面 99% 的内容和上次一样
        ↓
    只重新计算新敲的那 1 个字符 → 极快！
```

这就是为什么课件把"缓存机制"和"提示压缩策略"放在一起——都是为了**在毫秒级响应约束下，让大模型"用得起"**。

---

## 第 6 帧：和"提示压缩"是什么关系？

- **Prompt Cache**：避免重复计算相同内容
- **提示压缩**：把冗长的内容"瘦身"，让计算量更小

两者经常组合：先压缩，再缓存压缩后的结果。

---

## 去魔法化极简定义

> **Prompt Cache 就是把"已经处理过的提示文本对应的中间计算结果"存起来，下次遇到相同前缀时直接复用，避免重复计算。**

它不是什么魔法——就是一台"记忆缓存器"，把上次算过的中间结果原地存好，等下次同样的前缀再来时，直接拿出来用。**省的是重复计算，付的是显存/内存。**

---

## 简化边界

刚才这个比喻简化了几个真实细节：

1. **缓存粒度**：真实系统里缓存不是按整段对话存的，而是按"前缀匹配"的粒度（比如前 100 个 Token 一样就能复用）
2. **缓存失效**：如果历史对话被修改了（比如用户编辑了之前的消息），缓存就失效了
3. **实现位置**：缓存可以在客户端（本地）、服务端，或模型推理框架层（比如 vLLM、TensorRT-LLM）
