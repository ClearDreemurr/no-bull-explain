## T(n) = 4T(n/4) + n log₂n 的详细分析

> 用户回复："好，按照上面的节奏解释一下T(n) = 4T(n/4) + nlog_2 n"

### 低配场景

你有个长度为 n 的大任务。你的策略：

1. **把它砍成 4 段**，每段长度 n/4（a=4，b=4）
2. 每段派一个人递归去解决
3. 最后把 4 段结果拼起来——拼这一步要花 **n·log₂n** 的功夫

直观上你可能会想："拆成 4 段，每段 n/4，那合起来不也是 O(n) 级别的开销吗？"

但实际上你的 f(n) 是 **n log n**——比 n 大，但大得不够多。它正好卡在 Case 2 的尾巴上。

### 递归树展开

```
第 0 层:  [       整段长度 n       ]    花 n·log₂n
        ┌────┬────┬────┬────┐
第 1 层:[n/4][n/4][n/4][n/4]         4 段，每段花 (n/4)·log₂(n/4)
        ┌┬┬┐ ┌┬┬┐ ┌┬┬┐ ┌┬┬┐
第 2 层:...每段再拆 4 段...          16 段，每段花 (n/16)·log₂(n/16)
        ...
第 k 层:                             4^k 段，每段花 (n/4^k)·log₂(n/4^k)
```

**先看第 1 层总花销：**

4 段 × (n/4)·log₂(n/4) = **n·log₂(n/4)**

```
第 0 层:   n·log₂n            ← 根层
第 1 层:   n·log₂(n/4)        ← 少了 log₂4 = 2，比上层少了 n·2
第 2 层:   n·log₂(n/16)       ← 又少了 n·2
...
```

规律：**每下降一层，总花销减少 n·2**。

```
图表演示（只看数字趋势，n = 64 时）:

层 0:  n·log₂n        = 64·6  = 384
层 1:  n·log₂(n/4)    = 64·4  = 256
层 2:  n·log₂(n/16)   = 64·2  = 128
层 3:  n·log₂(n/64)   = 64·0  = 0      ← 到底了，log₄64 = 3 层
```

### 总花销

层数从 0 到 log₄n，每层花销是 n·(log₂n - 2k)，其中 k 是层数。

所有层加起来：

```
总和 = n·[  log₂n + (log₂n-2) + (log₂n-4) + ... + (log₂n - 2log₄n)  ]
    = n·[  log₂n + (log₂n-2) + (log₂n-4) + ... + 0  ]
```

共有 log₄n+1 项（≈ (log₂n)/2 项）。等差数列，每项差 2，首项 log₂n，末项 0。

等差数列求和公式：(首项 + 末项) × 项数 / 2

```
总和 ≈ n · (log₂n + 0) · (log₄n+1) / 2
     = n · log₂n · (log₂n / 2) / 2           ← log₄n = log₂n / 2
     = Θ(n · log²n)
```

### 映射回主定理

| 参数 | 值 | 角色 |
|------|----|------|
| a | 4 | 每层拆 4 个子问题 |
| b | 4 | 每个子问题缩到 1/4 |
| log_b a | log₄4 = 1 | n¹ = n |
| f(n) | n·log₂n | 等于 n¹·(log n)¹ |

**天平称量：**

- f(n) = n·log n
- n^(log_b a) = n¹ = n
- 比较：f(n) 比 n 大了一个 **log n** 因子，但没有大到差一个 n^ε

**定位：** Case 2 的变体，k=1（因为多了一个 log¹ n）

**结论：** Θ(n · log² n)

这是一个经典的 **Case 2 尾巴** 例子。f(n) 比 n 大又没那么大，结果既不是 Θ(n) 也不是 Θ(f(n))，而是中间的 **n·log²n**。

### 去魔法化定义

`T(n) = 4T(n/4) + n log n` 描述的是这样一类算法：拆成 4 等份去递归，合并成本是 O(n log n)。因为每层的总工钱形成一个等差数列递减，底层到 0，最后总工钱 ≈ 首项 n log n 乘以一半层数，结果是 Θ(n log² n)。

### 容易搞混的点

- 千万别觉得"拆 4 份就乘 4，合起来是 n log n"。要算的是**递归树上所有层的总花销**，不是单看任何一层。
- 这个式子也演示了为什么主定理要精确分三个 Case：如果只记"f(n) 大就 Θ(f(n))"，你就会得到错误的 Θ(n log n)——实际上它是 Θ(n log² n)，多了一个 log。

## Case 2 到 Case 3 的"生死线"在哪

> 用户回复："怎么判断这个"又没那么大"的边界？比如你给我举一个大而且"刚好有那么大"的例子"

这是主定理最反直觉的地方。用同一个 `a=4, b=4` 当固定参照物：

```
基线：n^(log_b a) = n¹ = n
```

现在让 f(n) 从 n 开始，一点一点往上"长胖"——看它什么时候脱离 Case 2、掉进 Case 3。

### 同框架下只换 f(n)

同样是 `a=4, b=4`，只换 f(n)：

```
递归结构固定不变:
    第 0 层:   全段 n        ← f(n) 在这里发生变化
    第 1 层:  4 段各 n/4     ← 4 × f(n/4)
    第 2 层: 16 段各 n/16   ← 16 × f(n/16)
    ...
```

**例子 A：f(n) = n**
```
第 0 层: n
第 1 层: 4 × (n/4) = n
第 2 层: 16 × (n/16) = n
...
每层都是 n → Θ(n log n)
```
Case 2，k=0，结论：**Θ(n log n)**

**例子 B：f(n) = n log n（你那个）**
```
第 0 层: n log n
第 1 层: 4 × (n/4)·log(n/4) = n·log(n/4) = n(log n - 2)
第 2 层: 16 × (n/16)·log(n/16) = n·log(n/16) = n(log n - 4)
```
等差数列递减 → Θ(n log²n)；**Case 2，k=1**

**例子 C：f(n) = n log² n**
```
第 0 层: n log² n
第 1 层: n (log n - 2)²
第 2 层: n (log n - 4)²
...
最终 → Θ(n log³ n)
```
**还是 Case 2，k=2**

### 核心发现——Case 2 的区间有多宽？

甚至 f(n) = n · log¹⁰⁰⁰ n，也**仍然是 Case 2**，结果是 Θ(n · log¹⁰⁰¹ n)。

**Case 2 的区间根本不是一条窄线，它是一个巨大的区间——所有「n 多乘几个 log」的场景全在里面。**

那什么时候才能跳到 Case 3？

**条件不是"f(n) 比 n 大"，而是"f(n) 比 n 大出来一个 n^ε"**，也就是多项式级别的差距。

**例子 D：f(n) = n¹·⁰⁰⁰¹**
```
n^(log_b a)  =  n¹
f(n)         =  n¹·⁰⁰⁰¹

差距因子 = n^0.0001        ← 这是一个多项式因子，不是 log 因子

→ Case 3
```

这种情况：
```
第 0 层:  n^1.0001
第 1 层:  4 × (n/4)^1.0001 = 4 × n^1.0001 / 4^1.0001
                              = n^1.0001 / 4^0.0001
                              ≈ n^1.0001 × 0.99986     ← 只缩了一点点
第 2 层:  ≈ n^1.0001 × 0.99972
...
```

每层缩减得非常慢，但总花销是几何级数求和：首项 n^1.0001 占绝对大头。

**结论：Θ(n^1.0001)**

### 一句话总结边界

```
f(n) 比 n^(log_b a) 大了……

      一个 log n        → 还是 Case 2
      一百个 log n      → 还是 Case 2
      甚至一万个 log n  → 还是 Case 2

      一个 n^0.0001     → Case 3 ← 跨线了！
```

**Case 1 ↔ Case 2 的边界同理：** f(n) 必须比 n^(log_b a) **小**一个 n^ε 的量级才算 Case 1。

> **主定理只认"多项式级别的差距"。用 log 量级堆得再高，也跳不出 Case 2 这个区间。**

## (log n)^1000 和 n^0.001 的比较

> 用户回复："那比较log(n)^1000和n^1.001呢"

这两个单拉出来比，答案很明确：**n^0.001 最终会赢**——但不是在你能见到的 n 范围内。

### 瘦身操作

```
我们比的是：      (log n)^1000   vs   n^0.001

两边同时开 1000 次方（取 1000 次根）：
       log n       vs   n^0.000001
```

问题变成：**log n 和 n^(10⁻⁶) 谁大？**

n^(10⁻⁶) 虽然指数极小极小，但它是**多项式**——而 log n 涨到后面会被任何一个正指数多项式踩在脚下。这只是时间问题。

### 数值推演

```
n = 10^6（一百万）：

  log₂n  ≈ 20
  (log n)^1000 ≈ 20^1000 ≈ 10^1300    ← 天文数字

  n^0.001 = (10^6)^0.001 = 10^0.006 ≈ 1.014    ← 几乎还是 1

  差距：10^1300 vs 1.014，log 赢。
```

```
n = 2^(10^8)（2 的一亿次方——这是一个 3000 万位的数字）：

  log₂n  = 10^8
  (log n)^1000 = (10^8)^1000 = 10^8000

  n^0.001 = 2^(10^8 × 0.001) = 2^100000 ≈ 10^30103

  差距：10^8000 vs 10^30103，n^0.001 反超！
```

(log n)^1000 一开始像火箭起飞，但它涨得**越来越慢**；n^0.001 刚开始像蜗牛，但它涨的**相对速度不会衰减**——这就是多项式 vs 对数幂的本质区别。

### 背后的数学真相

任何 log 的幂次，哪怕 (log n)¹⁰⁰⁰⁰，和任何 n^ε，哪怕是 n^0.0000001，的赛跑都是：

**数学规则：对于任何 ε>0 和任何 k>0，从某个 N 开始，n^ε > (log n)^k 永远成立。**

```
              ^
              |              n^0.001 从这里开始反超 ↑
  巨大值      |    ________________________/
             |   /
             |  /  (log n)^1000 在这里看起来很猛
             | /
             |/_____________________________→  n
                 交叉点 ≈ 10^(10^8)
```

### 回到主定理场景

回头看判断 Case 边界的条件，要"多项式地大于" n¹，也就是要 f(n)/n ≥ n^ε 对某个 ε>0 成立。

```
f(n)/n = (log n)^1000    ← 这个是和 n^ε 比

(log n)^1000  vs  n^ε    ← 就是我们刚才比的那一对
```

(log n)^1000 永远达不到 n^ε 的量级，因为它们是**不同类的怪物**。最终 n^ε 会把 (log n)^1000 碾过去。

所以：**就算 f(n) = n·(log n)¹⁰⁰⁰⁰，它也跳不出 Case 2。** 结果就是 Θ(n · log¹⁰⁰⁰¹ n)。

### 去魔法化表述

> 把 log 堆得再高——(log n)¹⁰⁰⁰、(log n)¹⁰⁰⁰⁰——都属于同一个 Case 2，结果只在 log 的指数上多 1。
>
> 但只要 f(n) 的 n 的指数从 1.0 变成 1.0000001，哪怕大了亿分之一，它就跨进 Case 3。

在算法复杂度的世界里，log 和 n^ε 的差距，比 n 和 2^n 的差距还本质。前者是"早晚会超过"，后者是"基本上没得比"。

## O(n log n) 和 O(n) 有什么区别？为什么"同一类"？

> 用户回复："哦哦懂了，但是O(nlogn)好像不能简单被认为是O(n)，这两个场景之间的区别是什么？"

### 两个同框架的对比

两个算法都符合 `T(n) = 4T(n/4) + f(n)`：

- **算法 A：f(n) = n**
- **算法 B：f(n) = n log n**

**先看每层的总花销各自怎么变：**

```
算法 A：每层花销
第 0 层:  n
第 1 层:  4 × (n/4)          = n
第 2 层:  16 × (n/16)        = n
第 3 层:  ...                = n
         → 每层完全相同         总和 ≈ n · log₄n = Θ(n log n)
```

```
算法 B：每层花销
第 0 层:  n·log n
第 1 层:  4 × (n/4)·log(n/4) = n(log n - 2)
第 2 层:  16 × (n/16)·log(n/16) = n(log n - 4)
第 3 层:  ...                = n(log n - 6)
         → 等差数列递减         总和 ≈ (首项 + 末项)×层数/2 = Θ(n log² n)
```

现在看到两件事：

- 算法 A：**每层一样多**，所以瓶颈均匀分布 → Case 2，Θ(n log n)
- 算法 B：**每层都在减少但没减到 0**，所有层加起来是个等差数列求和 → Case 2，Θ(n log² n)

两者真正的区别不是"谁到了哪个 Case"——**它们都在 Case 2 里**。区别是 Case 2 内部的 log 次数不同。

### 主定理划分 Case 的意义

**主定理只回答"瓶颈在哪一层"**，不回答"精确复杂度是多少"。

| 情况 | 瓶颈 | 总复杂度由谁决定 |
|------|------|-----------------|
| Case 1 | 最底层 | 叶子节点数量 `n^(log_b a)` 完全支配 |
| Case 2 | 均匀分布 | 所有层加起来，每层贡献平等 |
| Case 3 | 最顶层 | 根层 `f(n)` 完全支配 |

算法 A 和算法 B 的**瓶颈类型是相同的**——都是每层均匀分布（虽然 B 的每层在递减，但不是"一层占大头"）。而 Case 1 和 Case 3 则是**一层压倒性占主导**。

```
Case 1 / Case 3：一层占 >90% 的总工作量
                ┐
                │ → 其他层加起来不到 10%，可以忽略
                ┘

Case 2：所有层加起来，每一层都在公式的结果里留下了痕迹
                → 必须准确算出所有层的总和
```

这个差距才是主定理划分三个 Case 的根本原因——不是给出精确复杂度，而是**告诉你瓶颈的结构是什么**。

### 那"差不多"到底是什么意思？

**只在判断 Case 边界的意义上"归为一类"**，不是最终复杂度上相等。

换个角度想：主定理的长处是**排除两个极端**——判断"是不是叶子太爆炸了？"和"是不是根层太贵了？"——剩下的都丢进 Case 2。但丢进 Case 2 之后，你还是得自己动手算精确值：(n log n) 和 n 在 Case 2 内部的复杂度差了一个 log，**不是差不多，差很多**。

### 去魔法化表述

**Case 2 ≠ "所有 f(n) 差不多"；Case 2 = "瓶颈均匀分布在所有层，你得把每层加起来"。**

| 场景 | 该用什么判断 |
|------|-------------|
| 判断 Case 边界（1, 2, 3） | 只看「多项式量级」，log 不算数 |
| 算出 Case 2 的最终复杂度 | 每一层的 log 都要认真加起来 |

所以 n log n 在判断边界时"不算多项式大"——这个结论不影响你在 Θ(n log n) 里看到它比 Θ(n) 大的那一截。两个问题不矛盾，只是主定理只想回答第一个问题。
