## 先别管术语

想象你在考试。考官给你一本很厚的参考书，然后连续问你问题：

> "第 1 题：根据全书内容，分析第三章的主题思想。"

你翻开书，从第一页开始，**每一页都读一遍**，边读边标注哪些段落重要、哪些是废话。读完全书，你才开始写答案。这个过程很慢——因为你要把整本书从头到尾消化一次。

> "第 2 题：那第四章的主角和第三章有什么关系？"

现在问题来了：你是重新从第一页再读一遍整本书，还是……直接用刚才翻书时留下的标注？

## 朴素做法怎么坏

LLM 生成文本是**一个一个 token 往外蹦**的。假设你让它续写 "The cat sat on the"，它要生成下一个词。

**生成第 1 个 token 时**：模型把整个输入 "The cat sat on the" 从头处理一遍——每个词都算出它应该"关注"哪些其他词，算出每个词的 Key 和 Value（后文会讲这是什么）。最后产出第 1 个 token，比如 "mat"。

```
输入: [The] [cat] [sat] [on] [the]
       ↓     ↓     ↓     ↓     ↓
     全量计算：每个词都要算 Q、K、V，每个词都要看所有其他词
       ↓
输出: "mat"
```

**如果生成第 2 个 token 时也从头来一遍**：模型就得把 "The cat sat on the mat" 这 6 个词**全部重新算一次** Q、K、V。但——"The cat sat on the" 这 5 个词的 K 和 V，刚才不是已经算过了吗？一模一样的数据，为什么还算？

```
生成 token 1 时：
  [The] [cat] [sat] [on] [the]          ← 5 个词，全量算 K、V
  耗时：算 5 个词

生成 token 2 时（朴素做法）：
  [The] [cat] [sat] [on] [the] [mat]    ← 6 个词，全量重新算
  耗时：算 6 个词     ↑ 前 5 个和上一步一模一样！

生成 token 3 时（朴素做法）：
  [The] [cat] [sat] [on] [the] [mat] [is]  ← 7 个词，全量重新算
  耗时：算 7 个词     ↑ 前 6 个又重复算了一遍！
```

越往后越离谱——第 100 个 token 要重算前面 99 个。这就是**没有缓存的朴素做法**：每一步都把整条序列从头算一遍。

## 只看一个局部动作：K 和 V 是什么

先别管 Q、K、V 这三个字母。看一个具体的词——比如 "cat"——在 Attention 里做了什么。

```
"cat" 这个词要做两件事：

事 A：「我是谁，我有什么信息要说」
     → 这个是 V（Value），"cat" 携带的语义内容
     
事 B：「想找我的时候，用什么关键词能匹配到我」
     → 这个是 K（Key），"cat" 的索引标签
```

当后面的词——比如 "sat"——想知道自己应该关注谁时，它会发出一个 Q（Query）：「谁跟我最相关？」然后去跟每个词留的 K 做匹配，匹配度高的就拿那个词的 V。

```
"sat" 的 Q ──匹配──→ "cat" 的 K  ←──→ 匹配成功！
                              ↓
                    拿走 "cat" 的 V
```

---

用字符画看一次完整的生成过程。

**生成第 1 个 token（"mat"）**——必须从头算：

```text
步骤 1: 计算所有词的 K 和 V（存进缓存）
  The  → K₀, V₀
  cat  → K₁, V₁
  sat  → K₂, V₂
  on   → K₃, V₃
  the  → K₄, V₄
  └── 5 个词，5 次计算 ──┘

步骤 2: 最后一个词的 Q 去匹配所有 K
  Q₄ 去匹配 [K₀ K₁ K₂ K₃ K₄]
         ↓
  聚合对应的 V → 产出 "mat"

💰 成本：算 5 个 K,V + 1 次全量 attention
```

**生成第 2 个 token（"is"）**——有 KV cache：

```text
步骤 1: 只算新词 "mat" 的 K 和 V（前 5 个直接用缓存！）
  The  → K₀, V₀  ← 缓存
  cat  → K₁, V₁  ← 缓存
  sat  → K₂, V₂  ← 缓存
  on   → K₃, V₃  ← 缓存
  the  → K₄, V₄  ← 缓存
  mat  → K₅, V₅  ← 只算这一个！
  └── 只算了 1 个词 ──┘

步骤 2: "mat" 的 Q 去匹配所有 K
  Q₅ 去匹配 [K₀ K₁ K₂ K₃ K₄ K₅]
         ↓
  聚合对应的 V → 产出 "is"

💰 成本：算 1 个 K,V + 1 次增量 attention
```

---

停一下，确认你跟上了。

生成第 2 个 token 时，前 5 个词的 K 和 V 是算出来的还是从缓存拿的？

> 从缓存拿的

对。那继续看这个模式推下去会怎样。

**生成第 3 个 token（"soft"）**：

```text
  The  → K₀, V₀  ← 缓存
  cat  → K₁, V₁  ← 缓存
  sat  → K₂, V₂  ← 缓存
  on   → K₃, V₃  ← 缓存
  the  → K₄, V₄  ← 缓存
  mat  → K₅, V₅  ← 缓存
  is   → K₆, V₆  ← 只算这一个
  └── 新算 1 个词 ──┘
```

**生成第 100 个 token**：

```text
  前 99 个词 → K₀~K₉₈, V₀~V₉₈  ← 全部缓存
  第 100 个  → K₉₉, V₉₉         ← 只算这一个
```

你会发现一个规律：**不管序列多长，每生成一个新 token，都只算一个词的 K 和 V。** 前面所有词的 K、V 都存在显存里，用的时候直接拿。

## 成本对比

没有 KV cache 时：

```text
token 1: 算 5 个 K,V
token 2: 算 6 个 K,V（重算了前 5 个）
token 3: 算 7 个 K,V（重算了前 6 个）
...
token 100: 算 104 个 K,V（重算了前 103 个）
────────────────────────────
总计算量 = 5 + 6 + 7 + ... + 104 ≈ 5000+ 次计算
```

有 KV cache 时：

```text
token 1: 算 5 个 K,V
token 2: 算 1 个 K,V（前 5 个从缓存拿）
token 3: 算 1 个 K,V（前 6 个从缓存拿）
...
token 100: 算 1 个 K,V（前 103 个从缓存拿）
────────────────────────────
总计算量 = 5 + 1 × 99 ≈ 104 次
```

**5000 vs 104。** 这就是为什么第 2 个 token 开始突然"快很多"——不是第 1 个特别慢，而是从第 2 个开始，你只做了 1 个词的新计算，剩下的全是缓存读取。

---

## 代价是什么

缓存不是免费的。每生成一个 token，缓存就多存一组 K 和 V。100 个 token 的序列，显存里就躺着 100 组 K、V。这就是为什么 `GQA` 要共享 K、V 头、`MLA` 要压缩 K、V——**省的不是计算，是这块缓存的显存。**

---

## 去魔法化定义

> **KV cache 就是把每个词已经算过的"索引标签"（K）和"语义内容"（V）存在显存里，生成下一个词时只用算新词的，旧词的直接读缓存。省的是重复计算，付的是显存。**

这和考试翻书留标注是同一件事：第一遍翻书时留下的标注（K、V），让你后面每一题都只用在标注上检索，不需要重翻整本书。
