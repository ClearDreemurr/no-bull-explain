# 缓存与局部性原理详解 — 去魔法化逐步推演

> 为什么不是缓存越大越好？为什么要有 L1/L2/L3 三层？为什么改一行代码能快 100 倍？

---

## 一、先理解那个恐怖的落差

你在流水线上拧螺丝。你的手速：每秒能拧 10 颗。但传送带每隔 100 秒才送一颗过来。

```
时间轴（秒）：
  0s █ 拧一颗（0.1s）→ 然后干等 99.9 秒
100s █ 拧一颗（0.1s）→ 再干等 99.9 秒
200s █ ...
```

你的利用率：**0.1%**。

**CPU 和内存的关系，就是这个场景。** CPU 每秒执行几十亿条指令，但向内存要一个数据，要等几百个周期——中间 CPU 只能空转。

```
CPU 的视角：
[ 要读地址 0x1234 ]─────→ 内存
                        ↓ 等 200 个时钟周期…
[ 空转……什么都做不了 ]  ←──── 数据终于到了
```

这个落差，就是缓存存在的根本原因。

---

## 二、朴素解法：多存一点不行吗？

把一张小桌子放在手边，常用工具放桌上，取用时间从 100 秒降到 <1 秒。但**把小桌子换成大桌子，取东西会变慢**。

为什么？

### 原因 1：物理距离无法缩短

缓存芯片做得越大，物理面积越大。硅片上信号每走 1 厘米要多花约 50-60 皮秒（光速极限）。

```
芯片布局（俯视）：

小型缓存（快）：             大型缓存（慢）：
┌─────────┐                ┌─────────────────────────┐
│   CPU   │                │          CPU            │
│   ██    │                └─────────────────────────┘
└─────────┘                ┌─────────────────────────┐
    ↑                      │          缓存           │
    CPU 旁边就是缓存        │   最远那头离 CPU 很远    │
    (几毫米)               └─────────────────────────┘
                               ↑ 信号走到这里
                               需要多花几倍的延迟
```

**不是不想做大，做大了反而会慢，违背"快"的初衷。**

### 原因 2：寻址/译码电路随面积超线性增长

更大的缓存意味着更多的地址位需要译码，比较电路变得更长更复杂，进一步增加延迟。

## 三、分层绕过"大就慢"的矛盾

不做一个大缓存，做几个不同大小的，叠在一起。

```
                       CPU 核心
                       ┌────┐
        ┌──────────────┤ L1 ├──────────────┐
        │    取 1 个数据耗时：1 倍          │
        │    大小：~64KB                     │
        │              ┌────┐              │
        ├──────────────┤ L2 ├──────────────┤
        │    取 1 个数据耗时：5 倍          │
        │    大小：~512KB                    │
        │              ┌────┐              │
        ├──────────────┤ L3 ├──────────────┤
        │    取 1 个数据耗时：~15 倍        │
        │    大小：~8MB（但面积大，远）       │
        │              ┌────┐              │
        └──────────────┤内存├──────────────┘
            取 1 个数据耗时：~200 倍
            大小：几十 GB
```

### 一条数据怎么流动

```
位置：        CPU 先看 L1（64KB，紧贴核心）
              ↓
命中了（hit）→ 1 个周期读完，结束
              ↓
没命中（miss）→ CPU 立刻去问 L2，同时 L2 把这格数据搬回 L1
              ↓
命中了         → 5 个周期读完，下次就从 L1 读
              ↓
没命中         → 问 L3（8MB，芯片共享）
              ↓
命中了         → 15 个周期
              ↓
没命中         → 问内存 → 200 个周期
```

### 为什么分层有效

关键在两个字：**命中率**。

```
不命中的概率流（大致比例）：
访问 100 次数据 → 约 90 次在 L1（1 周期）
                → 约 8 次在 L2（5 周期）
                → 约 1.5 次在 L3（15 周期）
                → 约 0.5 次要闯内存（200 周期）

平均耗时 ≈ 1×0.9 + 5×0.08 + 15×0.015 + 200×0.005 ≈ 2.8 个周期
```

如果只用一个大 L3（没有 L1/L2），所有访问都要 15 周期。如果只用 L1（只有 64KB），命中率极低，大量访问落回 200 周期。

**分层让命中率和延迟达成了最佳妥协。**

## 四、缓存真正搬的是什么：Cache Line

CPU 从内存拿数据，不是"要一个字节就给一个字节"。它一次拿一整块，叫做 **Cache Line（缓存行）**，通常是 **64 字节**。

```
内存视角（每个方块 = 1 字节）：

地址 0x1000: [ A | B | C | D | E | F | G | H | … | … | … | … | … | … | … | … ]
              └───────────────── 64 字节 ─────────────────────┘
                                                ↑
                                CPU 要地址 0x1000 的数据，
                                内存一送就是整条 64 字节
```

读一个字节，后续 63 字节几乎免费。

```
CPU：我要读 0x1000
内存：好，给你 0x1000 ~ 0x103F 总共 64 字节

放进 L1 后：L1 现在有 [ 0x1000 ~ 0x103F ]

CPU：我要读 0x1001（紧挨着的下一个字节）
L1 翻了一下：我有这条 → 命中！→ 1 周期拿到

CPU：我要读 0x1002
L1：又命中！

……直到 0x103F 之前，全都不用再去内存。
```

## 五、局部性原理

两个核心规律让缓存有效工作。

### 规律 1：空间局部性（Spatial Locality）

**如果你用了地址 X，大概率过一会儿会用 X+1、X+2……**

```c
// 遍历一个数组
int sum = 0;
for (int i = 0; i < 1000; i++) {
    sum += arr[i];   // 按顺序访问 arr[0], arr[1], arr[2]...
}
```

```
Frame 1: CPU 要 arr[0]（地址 0x1000）
         └→ 内存搬进一条 cache line（0x1000~0x103F）
             L1 缓存现在有：[ arr[0] | arr[1] | arr[2] | … 共 16 个 int ]

Frame 2: CPU 要 arr[1]（地址 0x1004）
         已经在 L1 → 命中！1 周期

Frame 3: CPU 要 arr[2]（地址 0x1008）
         也已经在 L1 → 命中！

……后面 13 次访问全是命中，直到 arr[16] 触发下一条 cache line
```

**读一个 int，后面 15 个 int 都跟着被"顺带"预加载了。**

### 规律 2：时间局部性（Temporal Locality）

**如果你用了地址 X，很大概率过一会儿又要用 X 一次。**

```c
// 反复用同一个变量
int sum = 0;
for (int i = 0; i < 1000000; i++) {
    sum += 1;  // 每次循环都读写 sum
}
```

```
Frame 1: sum 在内存 → 被读进 L1，占 L1 的一格
Frame 2: 再读 sum → L1 里有 → 命中！1 周期
Frame 3: 又读 sum → 还在 L1 → 命中！
……
sum 在 L1 里住了一百万次，每次都是 1 周期
如果它不在 L1，每次就要 200 周期，慢了 200 倍
```

## 六、为什么改一行代码快了 100 倍

假设一个二维数组 `a[N][N]`，N=10000。

### 慢版本（按列遍历）

```c
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        sum += a[j][i];   // 内层 j 变 → a[j][i], a[j+1][i]……
    }
}
```

内存中数组按行存放（C 语言 convention）：

```
内存布局（一行接一行铺平）：
Row 0: [ a[0][0] | a[0][1] | a[0][2] | … | a[0][N-1] ]
Row 1: [ a[1][0] | a[1][1] | a[1][2] | … | a[1][N-1] ]
Row 2: [ a[2][0] | a[2][1] | a[2][2] | … | a[2][N-1] ]
```

慢版本访问顺序：`a[0][0] → a[1][0] → a[2][0] → …`

```
访问路径（慢版本）：
┌────────┐
│ a[0][0]│ ← 第一次，触发 cache line 加载 [a[0][0]~a[0][15]]
└────────┘       这行剩余的 a[0][1]~a[0][15] 全都用不上！
         ↓
┌────────┐
│ a[1][0]│ ← 第二次！a[1][0] 在另一行，地址跨了 N×4 字节
└────────┘   C A C H E   M I S S ！触发新 cache line
         ↓
┌────────┐
│ a[2][0]│ ← 第三次，又 miss！
└────────┘
```

**每次循环都 cache miss——上一行 cache line 里的数据全浪费。**

### 快版本（按行遍历）——只交换 i 和 j

```c
for (int i = 0; i < N; i++) {
    for (int j = 0; j < N; j++) {
        sum += a[i][j];   // 内层 j 变 → a[i][0], a[i][1], a[i][2]……
    }
}
```

```
访问路径（快版本）：
┌──────────────────────────────────────────────┐
│ a[0][0]│ a[0][1]│ a[0][2]│ … │ a[0][15]     │
│   ↑    │   ↑    │   ↑    │     │   ↑         │
│  1st   │  2nd   │  3rd   │     │  16th       │
└──────────────────────────────────────────────┘
第一次 a[0][0] 触发新 cache line，后续 15 次全命中！
          ↓
┌──────────────────────────────────────────────┐
│ a[1][0]│ a[1][1]│ a[1][2]│ … │ a[1][15]     │ ← 第 17 次触发新 cache line
│   ↑    │   ↑    │   ↑    │     │   ↑         │    然后免费落袋 15 次
└──────────────────────────────────────────────┘
```

### 量化对比

N=10000，总访问次数 1 亿次：

| | 慢版本（按列遍历） | 快版本（按行遍历） |
|---|---|---|
| 每次访问都 miss？ | **几乎每次** | 每 16 次才 1 次 miss |
| 总 miss 次数 | ~1 亿次 | ~625 万次 |
| 总周期（估算） | ~200 亿周期 | ~1 亿周期 |
| **倍率** | **~200×** | **1×（基准）** |

**同一行代码，只交换了 i 和 j——本质是把访问模式从"每次跳到远处"变成"顺着地址走"，让空间局部性发挥作用。**

## 七、术语映射

| 推演对象 | 术语 |
|---|---|
| CPU 一次从内存拿的 64 字节包裹 | **Cache Line（缓存行）** |
| 刚用过的东西过一会儿又要用 | **时间局部性（Temporal Locality）** |
| 用了一个东西，附近的东西很快也要用 | **空间局部性（Spatial Locality）** |
| 紧贴 CPU 的小桌子 | **L1 缓存** |
| 稍慢但更大 | **L2 缓存** |
| 更大更慢，芯片上共享 | **L3 缓存** |
| 利用这两种局部性的访问模式 | **缓存友好的代码（cache-friendly code）** |
| 把高频命中的数据放在小缓存里 | **缓存层级（Memory Hierarchy）** |

---

## 八、去魔法化极简定义

**缓存不是一种"存近一点"的魔法，而是一套用物理上更小、更近的存储，通过每次打包搬 64 字节的方式，赌你接下来会按顺序或重复访问的工程策略。**

- **不是越大越好**——大了就远了，远了就慢了
- **分层**——用"小且快"兜住高频命中、"大且慢"兜住剩余命中率
- **改一行代码快 100 倍**——本质是从"每次跳得远远的"变成"顺着地址访问"，把空间局部性用起来了

## 九、简化边界（省略了什么）

1. **缓存替换策略**——L1 满了后，新数据进来谁被踢走？（LRU、随机等）
2. **多核缓存一致性**——一个核改了 L1 里的数据，其他核的 L1 怎么办？（MESI 协议）
3. **硬件预取器**——现代 CPU 会自动识别连续访问模式，提前拉数据
4. **写策略**——写数据时是只写缓存（write-back）还是同时也写内存（write-through）
5. **TLB**——地址翻译也有缓存（快表），是另一个层面的局部性
