# 主定理（Master Theorem）讲解

## 第 1 帧：先别管公式和名字

想象你管一条生产线。你的工作方式永远一个套路：

**接到一个活 → 拆成几个小活 → 分给几个人干 → 最后把结果拼起来。**

而且拆完后，每个人拿到小活，也会继续用这个套路往下拆，直到拆成没法再拆的最小零件为止。

月底算总账时你最想搞清楚一件事：**我的时间到底花在哪了？**

是花在"拆来拆去"和"拼来拼去"那些 overhead 上了？还是花在最底层的"真正干活"上了？

这个问题，就是你眼前这个算法的总复杂度问题。

---

## 第 2 帧：只看一棵具体的「递归树」

拿 **二分查找** 举例。你在排好序的 8 个数里找目标：

```
第 0 层:   [1 | 3 | 5 | 7 | 9 | 11 | 13 | 15]     大小 n = 8
              ↓ 比 9，目标更大→扔掉左半边
第 1 层:         [9 | 11 | 13 | 15]                 大小 n/2 = 4
                     ↓ 比 13，目标更小
第 2 层:              [9 | 11]                      大小 n/4 = 2
                         ↓ 比 11，目标更小
第 3 层:                 [9]                        大小 1，搞定
```

每下一层，问题砍半。你**只追一个分支**，另一条路直接扔掉。

---

再看 **归并排序**——区别大了：

```
第 0 层:         [3|1|7|5|11|9|15|13]                   拆
               /                              \
第 1 层:  [3|1|7|5]                      [11|9|15|13]   2 个，各 n/2
          /          \                   /          \
第 2 层: [3|1]     [7|5]            [11|9]      [15|13]  4 个，各 n/4
         /  \      /  \             /   \        /   \
第 3 层: [3][1]  [7][5]         [11][9]    [15][13]     8 个，各 1
         \  /      \  /           \   /        \   /
第 4 层: [1,3]   [5,7]          [9,11]     [13,15]      合并
          \       /               \          /
第 5 层:   [1,3,5,7]           [9,11,13,15]             合并
               \                  /
第 6 层:      [1,3,5,7,9,11,13,15]                      合并完成
```

它**全拆全合**，每一层都在产生新的子问题，最后还要一步步合并回去。

---

**【停下确认 1】** 两个算法的关键区别在哪？

- 二分查找：每层只追 **1** 个分支，每个分支大小缩为 **1/2**，每层花 **O(1)** 时间做比较。
- 归并排序：每层拆成 **2** 个分支，每个大小缩为 **1/2**，每层花 **O(n)** 时间做合并。

这两个"参数"不同，总时间天差地别。**主定理就是专门回答"给定这三个参数，总复杂度到底是什么"的公式。**

## 第 3 帧：把它们参数化

任何刚才看到的那种"拆→分→拼"的算法，它的时间成本都可以写成：

```
T(n) = a · T(n/b) + f(n)
```

翻译成刚才的情景：

| 参数 | 一句话动作角色 | 二分查找 | 归并排序 |
|------|----------------|----------|----------|
| **a** | 每层拆出 **几个**子问题 | 1（只追一边） | 2（两边全追） |
| **n/b** | 每个子问题的规模缩到原来的 **几分之一** | n/2（砍半） | n/2（砍半） |
| **f(n)** | 拆完和拼起来这步，本身要花多少额外功夫 | O(1)（比一下） | O(n)（全合并） |

关键是：**a** 和 **b** 一起决定了递归树有多少叶子节点，而 **f(n)** 决定了每层非递归的工钱。

---

## 第 4 帧：现在比较这两笔账

拿归并排序看：

每一层的节点数（子问题个数）是 `a^k`（第 0 层 1 个，第 1 层 2 个，第 2 层 4 个……）。
每一层每个子问题的规模是 `n / b^k`。

把两者乘起来：**第 k 层所有子问题的总工作量** = `a^k · f(n / b^k)`

归并排序每层的合并成本加起来是 `O(n)`，然后有 `log₂n` 层，所以总 = `O(n log n)`。

但问题是——**不是所有算法都能这么简单地"层数 × 每层工作量"算出来**。因为 f(n) 和 a、b 之间的关系，会导致三种完全不同的结果：

1. **拆分的"分裂成本"特别高**（a 很大）→ 叶子层干活最多
2. **合并成本特别高**（f(n) 很大）→ 根附近干活最多
3. **两边差不多** → 每层均匀分摊

**主定理就是帮你判断：这三种情况，我遇到的是哪一种，然后直接告诉你答案。**

## 第 5 帧：三个比较对象

先记住：**主定理先把两个东西放在天平上称一称**。

| 称的左边 | 称的右边 |
|----------|----------|
| **f(n)** — 你这层拆和拼要花多少工钱 | **n^(log_b a)** — 递归树最底层有多少"叶子节点"在做工 |

> 注：n^(log_b a) 其实就是 a^(log_b n)，意思是"递归树到底层时，总共产生了多少个最小子问题"。

**三种结果 = 三种瓶颈位置。**

---

### Case 1：叶子太重（递归占大头）

> **条件：** f(n) 比 n^(log_b a) 长得慢，差了整整一个 n^ε 的量级。
> **结论：** Θ(n^(log_b a))

**直觉：** 上层拆分的开销还没铺开呢，底层已经被指数级增殖的子问题淹没了。

#### 例：T(n) = 4T(n/2) + O(1)

```
第 0 层:  [          全 1 段          ]  花 1
        /    |    |    \
第 1 层:[a][b][c][d]                   4 段各 n/2，每段花 1 → 共 4
       /|\  /|\  /|\  /|\
第 2 层:xxxxxxxxxxxxxxxxx              16 段各 n/4，每段花 1 → 共 16
       ...
第 k 层:                              4^k 段各 n/2^k，每段花 1 → 共 4^k
```

每层总花销爆炸性增长，最底层占绝对大头。

- a = 4，b = 2 → log₂4 = 2 → n²
- f(n) = 1，比 n² 慢得多（1 vs n²）
- **结论：Θ(n²)**

---

### Case 2：均匀分布（每层工钱一样多）

> **条件：** f(n) 和 n^(log_b a) 长一样快（差一个 log 因子以内）。
> **结论：** Θ(n^(log_b a) · log n)

**直觉：** 递归树每一层的总花销都差不多，层数 log n 层，所以乘个 log n。

#### 例：归并排序 T(n) = 2T(n/2) + O(n)

```
第 0 层:  [       合并花 n       ]    共 n
         /                   \
第 1 层: [ 合并花 n/2 ] [合并花 n/2]  共 n
        /     \          /     \
第 2 层:[n/4][n/4]    [n/4][n/4]      共 n
       ...
```

每层总花销都是 n，一共 log₂n 层 → Θ(n log n)

- a = 2，b = 2 → log₂2 = 1 → n¹ = n
- f(n) = n，和 n¹ 一样快
- **结论：Θ(n log n)**

---

#### 例：二分查找 T(n) = T(n/2) + O(1)

```
第 0 层:  [  比 1 次  ]             花 1
第 1 层:     [ 比 1 次 ]            花 1
第 2 层:        [比 1次]            花 1
               ...
```

每层花 1，一共 log₂n 层 → Θ(log n)

- a = 1，b = 2 → log₂1 = 0 → n⁰ = 1
- f(n) = 1，和 n⁰ 一样快
- **结论：Θ(log n)**

---

### Case 3：根太重（分裂/合并占大头）

> **条件：** f(n) 比 n^(log_b a) 长得快（差一个 n^ε），而且还有一条额外条件保证上层不会越拆越慢。
> **结论：** Θ(f(n))

**直觉：** 拆和拼的成本大得吓人，递归往下的成本根本比不过第一层。

#### 例：T(n) = 2T(n/2) + O(n²)

```
第 0 层:  [      合并花 n²       ]    共 n²
         /                    \
第 1 层: [ 合并花 (n/2)² ] [合并花 (n/2)²]  共 n²/2
        /     \              /     \
第 2 层:...总花销 ≈ n²/4 ...
```

第一层 n²，第二层 n²/2，第三层 n²/4……总和 ≈ 2n²，大头全在第一层。

- a = 2，b = 2 → log₂2 = 1 → n¹ = n
- f(n) = n²，比 n 快得多
- **结论：Θ(n²)**

## 第 6 帧：一张表总结

| 情况 | 天平偏向 | 瓶颈在哪 | 结论 |
|------|----------|----------|------|
| Case 1 | f(n) 远小于 n^(log_b a) | 叶子层子问题太多 | Θ(n^(log_b a)) |
| Case 2 | 平衡 | 每层均匀分布 | Θ(n^(log_b a) · log n) |
| Case 3 | f(n) 远大于 n^(log_b a) | 根层拆拼太贵 | Θ(f(n)) |

## 第 7 帧：去魔法化极简定义

**主定理：**
你有一个算法，每层拆成 `a` 个大小为 `1/b` 的子问题、额外花 `f(n)` 做拆拼。
比较"最底层子问题的指数级膨胀速度 `n^(log_b a)`"和"这层的拆拼开销 `f(n)`"谁大，
就能直接判定总复杂度。

它不是什么神秘定理——就是递归树的"哪一层是瓶颈"的判断工具。

---

## 第 8 帧：简化边界

- 这里讲的三个 Case 省去了一些边界条件和正则条件。真正的 Case 2 其实还细分了几个小分支（比如 f(n) 刚好是 n^(log_b a) log^k n 的变体），Case 3 需要验证 `a·f(n/b) ≤ c·f(n)` 才成立。
- 主定理**不能**处理所有递归形式。比如 `T(n) = 2T(n/2) + n log n` 就掉进了 Case 2 和 Case 3 之间的缝隙，需要用更精细的判断。
